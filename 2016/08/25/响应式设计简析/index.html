响应式设计简析

发表于 2016-08-25   |   分类于 移动端 // 插入位置

响应式设计初窥

响应式设计的主要目的是让网页适配当下种类繁多的设备，使页面在不同设备上看上去仍然友好可用

响应式中要面对的问题

像素密度(PPI)

PPI(Pixel Per Inch)直译为像素每英寸，表达的是在一英寸大小的方块中有多少个设备像素(物理像素)，也就是像素密度。

设备像素(物理像素) device pixel(physical pixel) : 与CSS像素px相对应，表达的是物理设备上的显示颜色的最小单位(设备上每一个点dot就是一个物理像素，所以PPI又称为DPI(dot per inch))

像素密度的计算公式：

对角线上的设备像素值/对角线的英寸长度

PS : 像素密度越高越好

BUT 单位面积内容纳的设备像素越多，每个设备像素的面积就越小，导致标清设备上的素材在高清设备上显示会缩小。

所以 高清设备上的素材应该具有 更多像素

在高清设备中，为解决设备像素过小的问题，会使一个像素对应多个设备像素，这个比值称为 设备像素比(DPR Device Pixel Ratio)

PS : 放大素材的前提是素材有足够的尺寸和像素

CSS像素(PX)

CSS像素默认与系统分辨率下像素大小相等。所以在标清设备中，一个CSS像素对应一个设备像素；在高清设备中，一个CSS像素可能等于多个设备像素

如果我们必须以设备像素进行开发，那么是非常繁琐的。因为每个设备的设备像素不一样，我们无法根据不同设备来提供如此多的素材。

所以我们需要一种抽象单位，只告知设备这个素材占据了多少抽象单位，它可以根据设备缩放到合适的大小，合适的具体的设备像素。这种抽象单位叫 设备无关像素

CSS像素就是一种设备无关像素。

所以我们不用关心在不同设备上一个CSS像素会匹配多少设备像素，设备会根据DPR在CSS基础上适当的放大缩小。

BUT 字体是矢量，拉伸后足够清晰；图片是标量，会模糊(因为一个CSS像素会横跨多个设备像素，信息被多个设备像素瓜分)

所以 需要为高清设备提供尺寸更大，细节更丰富的素材

PS : 高清素材可以向下兼容，可以用CSS像素控制高清素材在标清设备上的大小

视口

桌面浏览器视口

浏览器宽度等价于浏览器可视区域宽度(当html宽度为自适应的100%时)，可视区域大小为布局视口(layout viewport)简称视口(viewport)

BUT 当页面布局为固定宽度时，页面布局就不会受视口影响，视口以外的内容拖动滚动条可见

桌面端视口特点：浏览器区域受限于屏幕大小，页面宽度不会超过屏幕宽度

移动端浏览器视口

移动端浏览器打开页面步骤：

  1. 按照桌面端浏览器大小渲染页面(例：iOS4的Safari以980px渲染页面)
  2. 将页面缩放到宽度与系统宽度一致

PS : 用户手势缩放也是同样原理。并不是改变视口大小，而是改变视口缩放比(scale)

设计人员希望以页面的宽度来渲染页面，并自适应到系统宽度。所以手机厂商提供了一个name = “viewport”的meta标签，然后在content属性中设置参数来控制。

例：

设备宽度(device-width)

大多数情况下，我们希望以系统分辨率(视口大小)来渲染页面，尽可能的避免缩放.

此时，可以将width参数设置为device-width(设备宽度)，无论什么设备，什么握持方式，都按照系统分辨率来渲染

<meta name="viewport" content="width=device-width,initial-scale=1.0" />

问题：

  1. 当链接数目改变时，需要手动计算每个选项的宽度并进行更改
  2. 当桌面显示器较宽而链接数较少时，链接过于分散

第一个问题，可以用CSS3的calc()函数，让浏览器计算每个选项的宽度。calc()可以计算任何计算表达式， 即使表达式中减数和被减数使用不同的单位 比如calc(100% - 20px)。

第二个问题，给ul标签添加max-width属性来保证最大值不超过某个值

PS : 原始的width要与calc()同时存在，且width在前。这样可以保证当浏览器不支持calc()时可以使用width值；如果支持，那么calc()也可以将前面的width覆盖掉

问题：

BUT li不再平分父容器宽度(宽度不是确定值)，而又是向左浮动的，会造成导航栏向左偏移

子菜单

子菜单的实现就是在第一级的li标签中添加一个新的ul元素。子菜单相对于父容器li绝对定位，恰好在父元素下方(top:100%)，默认不显示

移动端

布局-响应式篇

媒体查询(media query)

media query 表达式包含两类查询条件

  1. 媒体类型(media type)

    在W3C最新标准中，有4种媒体类型：

    1. all(所有设备)
    2. print(打印设备，包括浏览器的打印预览信息)
    3. speech(能读出页面的设备，供残疾人使用)

    4. srceen(除打印和屏读以外的所有设备)

      PS : 如果没规定媒体类型，默认all

  2. 媒体特征表达式(media feature expression)

    包含在设备上验证的媒体特征，包含但不限于:width,max-width,min-width,device-width,aspect-ratio,color,resolution(分辨率),orientation:landscape(手机水平放置)

媒体查询既可以写在样式表中：

也可以写在样式外链<link>中

由于上线和性能的考虑，不会采用外链上增加媒体查询来加载针对性的样式，会增加额外的请求

媒体查询就是样式表中的if语句，也存在与、或、非来处理更复杂的情况

与(and)

使用and关键字来连接媒体类型和多个媒体特征，全部成立，样式才生效

或(,)

用逗号(,)来分割 多个媒体查询 任一样式成立都有效

非(not)

使用not关键字对当前媒体查询取反

例：not (max-width:600px) 只有max-width不成立，媒体查询才不成立

注意：

  1. 当not与and同时出现时，not是对整个媒体查询生效，而不是对距离最近的媒体特征生效
  2. not与逗号分隔多个媒体查询同时存在时，只对它所在的媒体查询生效，不影响其他查询

媒体查询的策略

  1. 顺序
    为了兼容多种设备，一般会提供多种媒体查询，此时顺序很重要。

    媒体查询匹配规则与样式表一致，从后往前进行匹配，一旦匹配成功立刻终止。

  2. max-width与min-width
    根据页面宽度由窄至宽划分(小于320px，320px-1024px，1024px以上)
    ，分别提供三类布局，一般是这样的顺序：

    但其实还可以更简洁

    这种方式叫移动优先(mobile-first)，页面优先采用移动样式，使用 min-width 来匹配宽度，随着宽度的增加样式逐渐向下匹配

    与之相反的叫桌面优先，页面优先采用桌面样式，用 max-width 来匹配宽度，随着宽度的减小样式逐渐向下匹配

    如果采用桌面优先，需要同时制定桌面和移动端的样式，而且为了保证移动端能够正常显示而不被桌面样式干扰，还需要对样式进行重置或覆盖。所以移动优先更具有优势。

相对单位 em

em是相对 最近父元素 的字体大小的相对单位。

比如body字体大小为16px，那么body中子元素字体1em大小即为16px

一种使用em单位的实践：先将body字体大小设置为默认的62.5%，前提是默认大小为16px，这样热河em想还原为px，只需要*10

相对单位 rem

rem单位与em单位类似，只不过rem相对于 根元素html ，也就是说只要是html内的任何元素，只要字体大小采用rem作为单位，那么都是以html字体大小作为参照，即使该元素不是根元素的直接子元素。

如何使用这几种单位

首先，设计仍然需要绝对单位，需要px。

实践思想：将页面分模块治之

  1. 将页面根据文字特征划分为几个模块，如导航、标题和正文
  2. 每一个模块的最外层的元素，如nav、header、article，都有自己的字体大小， 该字体大小以个元素的字体大小为准，用rem取值
  3. 模块最外层元素的字体大小，则是作为该模块内各元素字体大小的基准。

例如：设置<h1>元素的字体大小为2em，但在不同模块的h1大小是不一样的，根据模块调整字体大小。
